## Модул 1 от интеграционната архитектура

### Идентификатори и идентичност в системата gabriell‑e‑lit

Този документ дефинира всички правила за идентификатори, имена, идентичност и нормализация в системата gabriell‑e‑lit.
Той е фундаментален за:

- консистентност на данните
- автоматизация
- AI екстракция
- генериране на URL
- избягване на колизии
- стабилност на системата във времето

### 1. Основни понятия

#### 1.1. ID (уникален идентификатор)
Всеки обект в системата има уникален числов идентификатор.

Свойства:
- уникален в рамките на модела
- постоянен във времето
- не се променя при редакции
- не се използва за показване
- служи за вътрешни връзки между модели

Забрана:
- ID не се използва в URL
- ID не се използва като част от slug
- ID не се използва за SEO

#### 1.2. Canonical Name (канонично име)
Единственото име, което служи като идентичност на обекта.

Свойства:
- уникално в рамките на модела
- стабилно във времето
- не зависи от език
- не зависи от форматиране
- не се променя при редакции на display name
- използва се за генериране на slug
- използва се за автоматизация

#### 1.3. Display Name (име за показване)
Името, което се показва на сайта.

Свойства:
- може да бъде на български
- може да съдържа диакритики
- може да бъде поетично/стилистично
- може да се променя без да нарушава системата

### 2. Правила за идентичност

#### 2.1. Един обект = една идентичност
Независимо от език, превод, транслитерация или форматиране.

#### 2.2. Canonical name → slug → URL
Връзката е еднопосочна.

Display name не участва в тази верига.

#### 2.3. Canonical name не се променя
Дори при промяна на име, нов превод или ново издание.

#### 2.4. Колизии
При еднакъв canonical name се добавя дисамбигуиращ елемент:
- година
- място
- инициали
- номер

### 3. Нормализация на имената

#### 3.1. Български автори
Canonical name = оригинално име на кирилица.  
Display name = оригинално име на кирилица.  
Slug = латинизиран canonical name.

#### 3.2. Чуждестранни автори
Canonical name = оригинално име на латиница.  
Display name = оригинално име на латиница.  
Slug = латинизиран canonical name.

#### 3.3. Произведения на български
Canonical name = латинизиран вариант на българското заглавие.  
Display name = българското заглавие.  
Slug = латинизиран canonical name.

#### 3.4. Произведения на чужд език (оригинал)
Canonical name = оригинално заглавие.  
Display name = оригинално заглавие.  
Slug = латинизиран canonical name.

#### 3.5. Български имена — вътрешна латинизация gabriell‑e‑lit
Системата gabriell‑e‑lit **не използва официалната българска латинизация**, тъй като тя води до:
- загуба на идентичност
- колизии между фамилии
- абсурдни форми (напр. Balgariya)
- неразличими имена

Използва се вътрешна, стабилна, фонетично‑етична латинизация, която:
- запазва идентичността на автора
- е устойчива във времето
- избягва колизии
- е логична и предвидима

Примери:
- Габриела Цанева → Gabriela Tsaneva  
- Христина Въчева → Hristina Vacheva  
- Росица Вачева → Rositsa Vacheva  
- Георги Вучев → Georgi Vuchev  

#### 3.6. Произведения на чужд език, публикувани в превод на български
Ако произведението е публикувано в платформата в български превод:

- canonical_name = латинизиран вариант на българското заглавие  
- display_name = българското заглавие  
- slug = латинизиран canonical_name  

Пример:
Оригинал: The Red Orchard  
Публикувано като: Червената овощна градина  
canonical_name = "Chervenata ovoshtna gradina"  
slug = "chervenata-ovoshtna-gradina"

#### 3.7. Ръчен контрол върху canonical name и slug за автори
Идентичността на автора е човешка, не алгоритмична.

Ако автоматично генерираният canonical_name или slug не отразява правилно идентичността на автора,  
се използва **ръчно зададена стойност**, която става окончателна и стабилна.

Ръчният избор има приоритет над автоматичните правила.

#### 3.8. Display name на чуждестранни автори в преводни публикации

Display name на автора зависи от езика на публикацията.

1) Ако произведението е публикувано на оригиналния език:
- display_name = оригиналното име на автора (латиница)

2) Ако произведението е публикувано в превод на български:
- display_name = кирилизиран вариант на името на автора

Canonical name винаги остава оригиналното име на автора (латиница), независимо от езика на публикацията.

Slug винаги се генерира от canonical name, а не от display name.

Примери:
Оригинал:
canonical_name = "Theodore Smith"
display_name = "Theodore Smith"
slug = "theodore-smith"

Превод:
canonical_name = "Theodore Smith"
display_name = "Теодор Смит"
slug = "theodore-smith"


### 4. Идентификатори в JSON моделите

#### 4.1. Всеки модел съдържа:
- id  
- canonical_name  
- display_name (ако е приложимо)  
- slug  
- url  

#### 4.2. Връзки между модели използват ID
Никога не се използват имена, slug или URL.

### 5. Примери
(Примерите остават съкратени, защото идентичността е вече дефинирана в правилата.)

### 6. Роля в автоматизацията

#### 6.1. AI екстракторът използва:
- canonical name → идентичност  
- slug → URL  
- ID → връзки  

#### 6.2. AI никога не използва:
- display name  
- преводи  
- форматирани заглавия  

### 7. Резюме
- ID = вътрешна връзка  
- Canonical name = идентичност  
- Display name = визуално име  
- Slug = машинно име  
- URL = навигация  

## 📄 ОФИЦИАЛНА СЕКЦИЯ: ID ARCHITECTURE

### 8. ID Architecture

Тази секция дефинира официалната архитектура на идентификаторите (ID) в системата gabriell‑e‑lit.
Тя гарантира:

уникалност

стабилност

разширяемост

автоматизация

липса на колизии

предвидимост за AI екстрактора

ID е вътрешен технически ключ, който не се показва на сайта и не участва в slug или URL.

#### 8.1. Основни принципи
##### 8.1.1. ID е числов, уникален и постоянен
Всеки обект има един ID.

ID не се променя след създаване.

ID е уникален в рамките на модела.

ID не зависи от имена, slug, език или структура.

##### 8.1.2. ID не е публичен
Не се показва на сайта.

Не се използва в URL.

Не се използва за SEO.

Не се използва за визуализация.

##### 8.1.3. ID служи само за вътрешни връзки
Пример:

Код
publication.authors = [100123, 100456]

#### 8.2. Диапазонна архитектура
Всеки модел получава собствен диапазон от 100 000 ID, което гарантира:

липса на колизии

лесна автоматизация

ясна валидация

възможност за бъдещо разширяване

независимост между моделите

##### 8.2.1. Официални диапазони
Модел	Диапазон	Размер
Authors	100 000 – 199 999	100 000
Publications	200 000 – 299 999	100 000
Paintings	300 000 – 399 999	100 000
Collections	400 000 – 499 999	100 000
Exhibitions	500 000 – 599 999	100 000
Magazine Issues	600 000 – 609 999	10 000
Magazine Covers	610 000 – 649 999	40 000
Tags	800 000 – 899 999	100 000
Gallery Assets	900 000 – 999 999	100 000


##### 8.2.2. Защо диапазоните са по 100 000
покриват реалните нужди на системата

позволяват растеж за десетилетия

не са прекомерни

лесни са за автоматизация

позволяват добавяне на нови модели без промени в старите

#### 8.3. Добавяне на нов модел
Ако системата се разширява (например „audio works“), новият модел получава нов диапазон, например:

Audio Works → 1 000 000 – 1 099 999

Няма ограничение в броя модели.
Няма ограничение в броя диапазони.
Няма нужда от преномериране.

#### 8.4. Правила за генериране на ID

##### 8.4.1. Ръчно генериране
При създаване на нов обект се избира следващият свободен ID в диапазона.

##### 8.4.2. Автоматично генериране
AI екстракторът може да:

открива последния използван ID

генерира следващия

валидира, че ID е в правилния диапазон

открива колизии

##### 8.4.3. Забранено
използване на ID извън диапазона

използване на ID от друг модел

използване на ID като slug или URL

#### 8.5. Валидация

##### 8.5.1. Вътрешна валидация
Всеки JSON файл трябва да съдържа:

id

canonical_name

slug

url

##### 8.5.2. Диапазонна валидация
Ако ID е извън диапазона → грешка.

##### 8.5.3. Връзкова валидация
Всички ID във връзки (напр. authors, paintings, tags) трябва да:

съществуват

бъдат валидни

бъдат в правилния диапазон

#### 8.6. Примери

##### 8.6.1. Автор
Код
{
  "id": 100123,
  "canonical_name": "Gabriela Zaneva",
  "display_name": "Габриела Занева",
  "slug": "gabriela-zaneva",
  "url": "/authors/gabriela-zaneva/"
}
##### 8.6.2. Публикация
Код
{
  "id": 200044,
  "canonical_name": "Pismo do badeshteto",
  "display_name": "Писмо до бъдещето",
  "slug": "pismo-do-badeshteto",
  "authors": [100123],
  "url": "/publications/pismo-do-badeshteto/"
}

#### 8.7. Резюме
ID е вътрешен технически ключ

диапазоните гарантират стабилност

всеки модел има собствен блок

новите модели получават нови блокове

AI екстракторът работи оптимално с тази архитектура
