# validation.md

## Правила за валидация на данните в платформата gabriell‑e‑lit

### 1. Цел на документа
Този документ дефинира официалните правила за валидация на всички данни, използвани в платформата gabriell‑e‑lit.  
Валидацията гарантира:

- структурна коректност  
- уникалност  
- предвидимост  
- съвместимост с автоматизацията  
- устойчивост на системата във времето  

Всички JSON модели, slug‑ове, canonical имена и връзки между обекти трябва да отговарят на тези правила.

---

## 2. Валидация на JSON моделите

### 2.1. Структурна валидация
Всеки JSON файл трябва да бъде:

- валиден JSON (без trailing commas, без незатворени скоби)  
- форматиран с отстъпи (2 или 4 интервала, но последователно)  
- структуриран според модела в `/models`  

### 2.2. Задължителни полета
Всеки модел има задължителни полета, описани в съответния JSON модел.  
Примери:

- author.json → `canonical_name`, `display_name`, `slug`, `id`  
- publication.json → `canonical_title`, `slug`, `authors`, `id`  

Липсващо задължително поле = невалиден модел.

### 2.3. Типове данни
Типовете трябва да са:

- string  
- number  
- boolean  
- array  
- object  

Неподходящ тип = невалиден модел.

---

## 3. Валидация на идентичността

### 3.1. canonical name
Canonical name трябва да бъде:

- стабилен  
- уникален  
- в оригиналния език  
- без HTML  
- без специални символи (освен ако са част от името)  

### 3.2. display name
Display name трябва да бъде:

- визуално коректен  
- в езика на интерфейса (обикновено български)  
- без латинизация  
- без slug елементи  

### 3.3. ID
ID трябва да бъде:

- уникален в рамките на типа  
- стабилен  
- непроменяем  

ID не се генерира от slug.

---

## 4. Валидация на slug

### 4.1. Основни правила
Slug трябва да бъде:

- латинизиран  
- в малки букви  
- разделен с тирета  
- без специални символи  
- без стоп‑думи  
- без кирилица  

### 4.2. Уникалност
Slug трябва да бъде уникален в рамките на типа:

- два автора не могат да имат един и същ slug  
- две публикации не могат да имат един и същ slug  

### 4.3. Колизии
При колизия се добавя минимален числов суфикс:

ivan-vazov-2


---

## 5. Валидация на URL

### 5.1. Съвместимост със slug
URL трябва да бъде:

- директно производен от slug  
- без допълнителни символи  
- без автоматично добавени параметри  

### 5.2. Структурна коректност
URL трябва да следва правилата от `urls.md`.

---

## 6. Валидация на връзките между модели

### 6.1. Автори → Публикации
Всяка публикация трябва да съдържа:

- масив `authors`  
- всеки елемент трябва да бъде валиден ID на автор  

### 6.2. Публикации → Списание
Ако публикацията е част от брой:

- `issue_id` трябва да съществува  
- брой с такъв ID трябва да е валиден  

### 6.3. Галерия → Активи
Всеки gallery_asset трябва да има:

- валиден `id`  
- валиден `slug`  
- валиден `canonical_name`  

---

## 7. Валидация на SEO блокове

### 7.1. Заглавие
SEO заглавието трябва да бъде:

- кратко  
- ясно  
- без повторения  
- без slug елементи  

### 7.2. Описание
SEO описанието трябва да бъде:

- между 120–160 символа  
- без HTML  
- без специални символи  

---

## 8. Примери

### 8.1. Валиден author.json

{
"id": 12,
"canonical_name": "Иван Вазов",
"display_name": "Иван Вазов",
"slug": "ivan-vazov"
}

### 8.2. Невалиден slug

"slug": "Иван-Вазов"


Причина: кирилица.

---

## 9. Резюме
Валидацията гарантира, че всички данни в платформата gabriell‑e‑lit са:

- структурно коректни  
- уникални  
- стабилни  
- предвидими  
- съвместими с автоматизацията  

Тези правила са задължителни за всички JSON модели, slug‑ове, canonical имена, URL‑и и връзки между обекти.
